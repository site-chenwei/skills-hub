# 审查维度

## 功能正确性

- 是否存在逻辑错误或漏分支？
- 是否改变了旧行为但没有同步更新调用方或文档？
- 是否有边界条件、空值、并发或顺序问题？
- 当前代码是否能解释新增分支、默认值或状态转换的正确性？

## 契约一致性

- 接口、配置、数据结构、持久化格式是否兼容？
- 错误码、返回值、事件格式是否保持一致？
- 上下游是否会因为默认值变化而回归？
- 迁移、生成代码或 schema 变化是否同步覆盖消费者？

## 验证充分性

- 现有测试是否覆盖受影响路径？
- 关键失败路径是否被验证？
- 如果没有自动化测试，是否说明了原因和替代验证方式？
- 变更触及构建、依赖或配置时，是否还有构建/集成/手工验证空白？

## 非功能风险

- 性能：是否引入重 IO、重内存或额外 N+1？
- 安全：是否暴露凭据、未校验外部输入、扩大权限？
- 稳定性：是否吞错、静默降级、留下不可观测失败？
- 发布链路：CI/CD、构建脚本、锁文件或镜像是否可能引入隐性回归？
