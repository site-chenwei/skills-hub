# 排障清单

## 先确认

- 失败命令或操作步骤是什么？
- 失败是稳定复现还是偶发？
- 最近改动过哪些文件、依赖、配置、环境变量？
- 失败发生在构建、启动、运行时、网络交互还是持久化阶段？
- 是否已经用 `scripts/capture_failure.py` 固定住失败命令、退出码和日志尾部？

## 观察与定位

- 读完整错误栈，不截断关键信息。
- 确认断点在哪一层，不跨层乱修。
- 只加必要的日志或断言，修完后清理。
- 先把第一处真实失败点和后续连锁报错区分开。

## 修复与验证

- 优先根因修复，不用吞错掩盖。
- 每次改动后重跑最小充分验证。
- 若验证失败，继续定位，不叠加第二个猜测修复。
- 高风险问题在最小验证通过后，要补跑受影响范围内更广的验证。

## 外部阻塞

出现以下情况时要明确写成阻塞，而不是继续猜修：

- 外部服务未启动、无权限、凭据缺失、端口被占用
- 本地环境与项目声明的工具链版本不一致
- 失败只在特定机器或 CI 环境出现，当前机器无法复现对应前提
