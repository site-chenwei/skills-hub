# 项目入口检查清单

## 推荐顺序

1. 先跑 `<python_cmd> <skill_root>/run.py project_facts ...` 拿到仓库事实草图。
2. 读 `PROJECT.md` 与 `README*`，确认项目目标和任务上下文。
3. 读顶层配置与清单文件，确认技术栈、包管理器、测试入口、运行前提。
4. 只在当前任务需要时补读 `docs/`、关键源码与测试。

## 至少要回答的问题

- 项目的核心目标和主要交付物是什么？
- 本次任务期望的结果、明确不做什么、是否有范围外限制？
- 哪些文件、模块、配置、接口和数据结构会受影响？
- 当前仓库的技术栈、包管理器、测试框架和运行环境是什么？
- 最小充分验证命令是什么，是否还需要构建、集成或手工验证？
- 哪些结论是已确认事实，哪些只是基于文件信号的推断？

## 事实层级

- 已确认：
  来自存在的文件、配置、命令输出或已读源码。
- 推断：
  来自技术栈信号、目录命名、脚本命名或约定俗成的验证方式。
- 待确认：
  缺失会改变执行路径、验证策略或影响范围的问题。

## 高风险触发器

出现以下任一项时，不要只看 README：

- 多个顶层应用或多个包管理器并存
- 同时存在前后端 / 服务端 / 基础设施目录
- 配置、schema、迁移、依赖或生成代码目录参与本次任务
- 仓库没有明确测试入口，或测试/构建脚本与常规约定不一致
